   三、修订后的分期方案

   P0 —— 正确性（小改动，立即做）

   P0-1 缓存 Key 版本化（Gap 1）
   • 落点：lib/sw/constants.js 加 PROMPT_VERSION，cache-keys.js 的 generateCacheKey 编入 key。
   • 上次遗留的决策点仍待拍板：是否编入 model id。我的建议维持不变——编入。档案切换是高频操作，不同模型译文混用同一缓存会让"切档案对比效果"这个功能失效（看
     到的可能是旧模型缓存）。
   • 顺手清掉 spinner 残骸（content.js:134、content.css:355）。

   P0-2 用户取消链路（Gap 2）
   • CS 恢复原文/关闭翻译时发 cancel；SW 维护 taskId → AbortController 映射，复用现有超时 controller 模式（background.js:1616 等四处）扩展。动态内容翻译
     （_processAddedNodes）尤其需要——用户停止整页翻译后，在途的动态批次应中止而不是继续烧配额。

   P1 —— 调度与视口（体验主力）

   P1-1 视口感知（Gap 3）
   • 落点：content.js 整页翻译 + 新动态监听两处。段落块注册 IntersectionObserver（rootMargin: '200px'），入视口才提交；离视口未译的可取消/降级。
   • 与 P1-2 已完成部分的关系：动态 observer 解决了"新增内容翻不翻"，viewport 解决"已有内容什么时候翻"，两者正交互补。这是设计文档首段延迟 P50 ≤800ms 目标
     的核心手段。
   • 注意：现有"首屏 mini-batch 优先"逻辑应被此机制取代而非叠加，避免两套优先级打架。

   P1-2 优先级队列 + in-flight 去重（Gap 4）
   • 落点：SW 侧新模块 lib/sw/scheduler.js——这同时是拆分 3160 行 monolith 的自然第一步。
   • 三级优先级：视口内 > 预加载区 > 手动全文/动态批次。相同 cacheKey 的并发任务合并为共享 Promise 广播结果（整页 + 动态监听 + 划词三路并发时这个收益实际不
     小）。
   • 与现有自适应限速器的关系：限速器管"发多快"，调度器管"先发谁"，组合不冲突。

   P2 —— 精准度

   P2-1 段落级滑动窗口（Gap 5）
   • 在现有 buildTranslationPrompt 的 context 参数上扩展：批量分块时携带前一块末尾 1–2 句"原文+译文"作 <context> 参考（标记勿重译）。改动集中在
     translate-core.js + 调用处，成本低，文档实测指代准确率 83%→96%。

   P2-2 内联标签占位符保护（Gap 6）
   • 提取阶段将 <a><b><code> 等替换为 <t n="1">，回收按序号还原 DOM。改动面最大、回归风险最高，建议独立成项，用维基百科长条目/GitHub README 手动回归。前置
     条件：P1 稳定后再动提取层。

   P3 —— 架构债（完成 P0–P2 后评估）

   P3-1 引擎抽象层 + SW 拆分：供应商配置表 → Translator 接口 + lib/sw/engines/*.js 适配器（文档"新增引擎 ≤50 行"目标）；background.js 按
   scheduler/cache/engines/rate-limiter/消息路由继续拆分。3160 行的 monolith 是目前可维护性的最大短板，但拆分本身不产生用户价值，放最后。

   维持不采纳（结论不变）

   TS+esbuild 重构、Shadow DOM、Google 免费引擎兜底、Firefox、storage.sync——理由同上一版，现状没有让这些结论变化的新因素。

   四、落地顺序

   1. P0（key 版本化 + 取消链路 + spinner 清理）——一个迭代内完成；
   2. P1-1 视口感知 → P1-2 调度器（顺序可调换，但调度器涉及 SW 拆模块，工作量略大）；
   3. P2-1 滑动窗口（可提前，改动小）→ P2-2 占位符（独立回归）；
   4. P3 视精力，不阻塞发布。
