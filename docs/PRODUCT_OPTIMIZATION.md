## 1. 产品现状一句话

**能用、功能厚、上手偏难、设置与实现偶有脱节。**  
对「已配置好 API / Ollama 的深阅读用户」体验不错；对「刚装上想立刻划词翻译」的新用户不够友好。

| 版本与体量 | 数值 |
|------------|------|
| 版本 | `manifest` 0.4.1（随本方案 Phase A–D 发布） |
| 核心代码 | `background.js` ~2.5k 行 / `content.js` ~1.1k / `options.js` ~1.1k |
| 测试 | `background.test.js` 22 项通过；无 content/popup E2E |
| 主路径 | 划词浮窗 → 流式单句；整页 → 去重 + mini-batch 首屏 + 批量后续 |

## 2. 代码审视结论

### 2.1 做得好的地方

| 能力 | 评价 |
|------|------|
| 缓存质量校验（ADR-0004） | 严格键 + 长度门槛 + 坏命中拦截，质量优先正确 |
| 整页翻译管线 | 可视区优先、去重、批量降级、失败标记、可恢复原文 |
| ProviderProfile | 多档案切换，符合「控制面板」定位（ADR-0001） |
| 自适应限流 | 429 冷却、并发上下调，有生产意识 |
| IndexedDB 重连雏形 | `openDatabase` 事务探活 + `onclose` 重置 |
| 诊断与用量 | 请求日志、metrics、token 估算，便于排障 |
| 视觉系统 | 书房衬纸 tokens，三端统一 |

### 2.2 问题清单（按用户可感知严重度）

#### P0 — 直接伤核心体验

| ID | 问题 | 证据 / 影响 |
|----|------|-------------|
| C1 | **划词触发模式配置未生效** | 设置有「选中后弹出 / 悬浮图标 / 右键」；`content.js` 始终 `showFloatButton`，未读 `triggerMode` |
| C2 | **流式开关未真正贯通 content** | Popup 有流式开关；划词固定走 `translateStream`，未读 `enableStreaming` |
| C3 | **零配置冷启动空窗** | `onInstalled` 只建菜单，无引导；未配置时错误文案有，但缺「一键打开设置」主路径 |
| C4 | **文案失实** | 设置写「API Key 将安全加密存储」——实际为 `chrome.storage` 明文；削弱信任 |
| C5 | **版本号不一致** | `options.html` 硬编码 `v0.3.0`，manifest 为 `0.4.0` |

#### P1 — 产品摩擦与可靠性

| ID | 问题 | 影响 |
|----|------|------|
| C6 | 设置信息架构偏「工程师控制台」 | 三栏齐全但首屏即供应商细节；诊断、缓存 MB、黑白名单同级展示，学习成本高 |
| C7 | Popup「热词数」与空 `BUILTIN_CACHE` 矛盾 | 统计项对用户无意义甚至误导 |
| C8 | 短文不缓存（≥12 字）正确但未解释 | 用户感觉「划词老是打 API」，缓存命中率体感差 |
| C9 | 站点黑名单无法在页面内快速禁用 | 必须进设置改域名列表 |
| C10 | SW 休眠后偶发「第一次失败、第二次成功」风险 | 已有 DB 探活，但消息层无统一 `ensureReady` + 用户侧柔和重试提示 |
| C11 | 错误体系半成品 | 有 `formatError`，无统一错误码结构；网络失败 / Ollama 未启动等文案仍偏技术 |
| C12 | 巨型单文件 | 难测、难审、难并行改功能；测试只覆盖 background 纯函数 |

#### P2 — 场景缺口（非立刻阻塞）

| ID | 问题 |
|----|------|
| C13 | 无「本页重新翻译 / 仅译选区段落」细粒度控制 |
| C14 | 无用户术语表 / 固定译名 |
| C15 | 无译文反馈（差译标记）闭环到缓存淘汰 |
| C16 | 语言列表不全（通用设置目标语偏少） |
| C17 | 无构建与 CI；商店发布前难回归 |

### 2.3 架构简图（现状）

```
用户动作
  ├─ 划词 → content 浮钮 → translateStream ──┐
  ├─ 快捷键/右键 → bg → content ─────────────┤
  └─ Popup「整页」→ content.translatePage ───┤
                                              ▼
                                    background.js（全能）
                                      ├ 配置 / 档案
                                      ├ 缓存 + 校验
                                      ├ 批量 / 流式 API
                                      └ 限流 / 指标
```

瓶颈不在「功能有没有」，而在：**入口体验、配置生效、可理解反馈、代码可维护性**。

## 3. 使用者视角：关键旅程与痛点

### 3.1 新用户（第一次安装）

1. 加载扩展 → 点图标 → 看到「请先在设置页添加模型」  
2. 进设置 → 面对服务商 / Key / 端点 / 档案 / Ollama 推荐  
3. 填完保存 → 回网页划词才第一次成功  

**痛点**：缺少 30 秒「最短成功路径」；本地与云端两条线同等复杂。

**期望**：安装后 3 步内完成第一次成功翻译。

### 3.2 日常深阅读用户

- 划词看释义、整页双语、偶换模型  
- **痛点**：整页慢时不知是缓存/限流/模型；失败节点无「重试此句」；PDF 站、SPA 动态内容体验不稳定  

### 3.3 隐私敏感用户（本地 Ollama）

- 推荐轻量模型路径已存在  
- **痛点**：Ollama 未启动时提示可再软；缺「离线模式 = 禁止云端」硬开关  

### 3.4 高级用户 / 调试者

- 诊断页、请求日志已够用  
- **痛点**：与普通用户设置混在一起，显得产品「重」

## 4. 产品经理视角：定位与原则

### 4.1 产品定位（收敛）

**YuxTrans = 面向深阅读的网页 AI 翻译扩展**  

- 不做：桌面客户端、Python SDK、输入框式独立翻译器（ADR-0001 已否决）  
- 核心价值顺序：**准 → 稳 → 快 → 轻**（与 slogan 一致，但准优先于炫技功能）

### 4.2 产品原则

1. **配置一次，处处生效** — 设置项必须真正驱动行为（C1/C2）  
2. **默认正确** — 默认流式、双语、自动语言检测；高级项折叠  
3. **失败可理解、可行动** — 每条错误带「下一步」  
4. **缓存诚实** — 质量校验策略保留，但要在 UI 说清「为何没命中」  
5. **工程师能力藏在诊断里** — 普通用户不看 metrics / request logs  

### 4.3 成功指标（建议追踪）

| 指标 | 目标（方向） |
|------|----------------|
| 安装 → 首次成功翻译时间 | &lt; 2 分钟 |
| 划词成功率（排除用户取消） | &gt; 95% |
| 整页「首屏有译文」时间 | 可感知 &lt; 3s（视模型） |
| 坏缓存命中率（已有 blockedHits） | 持续下降 |
| 因「不知道怎么配」卸载/差评 | 趋近 0 |

## 5. 优化方案（分阶段）

### Phase A — 信任与正确性（1 周，必做）

> 目标：设置即生效、文案诚实、版本一致、冷启动可走通。

| 项 | 动作 | 对应问题 |
|----|------|----------|
| A1 | `content.js` 读取并遵守 `triggerMode`：`auto` 直接译、`icon` 浮钮、`contextMenu` 不弹钮 | C1 |
| A2 | 划词 / 整页尊重 `enableStreaming`：关则走 `translate` | C2 |
| A3 | 首次安装：`onInstalled` 打开 options 或轻量欢迎页（3 步：选本地/云端 → 填 Key 或检测 Ollama → 试译一句） | C3 |
| A4 | 去掉「加密存储」措辞，改为「仅保存在本机浏览器，不会上传到 YuxTrans 服务器」 | C4 |
| A5 | 版本号统一读 `manifest.version`；修 options 硬编码 | C5 |
| A6 | Popup 无档案时主按钮改为「去配置翻译服务」并 `openOptionsPage` | C3 |
| A7 | 去掉或改写「热词数」为「缓存条目」等真实指标 | C7 |

**验收**：改设置后无需猜；新装用户有明确引导；无虚假安全承诺。

---

### Phase B — 体验打磨（1–2 周）

> 目标：日常好用、失败不慌、整页可控。

| 项 | 动作 | 对应问题 |
|----|------|----------|
| B1 | 统一错误对象：`code` + `userMessage` + `actionHint`（如「打开设置」「检查 Ollama」） | C11 |
| B2 | 整页控制条：失败句一键重试；显示「缓存命中 x / API y」 | C8 部分 |
| B3 | 页面内「本站禁用 YuxTrans」快捷入口（写入 siteList） | C9 |
| B4 | 设置 IA：默认「快速开始」+ 折叠「高级」；诊断保持独立 Tab | C6 |
| B5 | 缓存策略说明：短文不缓存的一句话 hint；命中失败不静默 | C8 |
| B6 | SW 消息入口 `await ensureInitialized()`；DB 失败时重试 1 次并 toast「服务恢复中」 | C10 |
| B7 | 语言列表补齐常用目标语（与右键菜单一致） | C16 |

**验收**：主路径错误用户能自救；整页失败可局部重试。

---

### Phase C — 质量与深度阅读（持续）

> 目标：从「翻得出」到「翻得好、记得住」。

| 项 | 动作 |
|----|------|
| C-a | 用户术语表（CSV/JSON 导入，翻译前强制替换） |
| C-b | 差译反馈 → 剔除该缓存键 + 可选不上报原文 |
| C-c | 按站点记住双语/仅译文偏好 |
| C-d | 离线模式硬开关（仅 local + cache） |
| C-e | 可选：悬停释义模式（不改 DOM 结构，适合文档站） |

---

### Phase D — 工程化（与功能并行）

| 项 | 动作 |
|----|------|
| D1 | 拆分 `background.js`：`cache` / `providers` / `translate` / `messages`（可先无构建，多 script 或简单 concat） |
| D2 | 为 `content` 关键路径加可测纯函数 + 扩展测试用例 |
| D3 | 最小 CI：`node --test extension/tests/` |
| D4 | 发版检查清单：版本三处一致、CHANGELOG、zip 打包 |

## 6. 明确不做（本周期）

- 恢复桌面端 / Python 包作为产品主线  
- 为「热词预热」回退 ADR-0004 严格缓存（质量优先）  
- 重型前端框架 / 完整 Vite 工程（除非 D1 拆分受阻）  
- OCR、PDF 专用引擎、语音翻译  
- API Key 主密码加密（可进 Phase C 隐私增强，非 A/B 阻塞）

## 7. 建议执行顺序（两周切片）

```
Week 1:  A1–A7（正确性 + 冷启动） → 手工验收主路径
Week 2:  B1–B4 + B6（错误、整页、设置 IA、SW 就绪）
之后:    B5/B7 → C-a/C-b → D1/D3
```

优先级口诀：

> **先让配置生效，再让新手成功，再让失败可懂，最后才加新功能。**
